# Analysis Plan — Інваріантність ознак продуктивності гравців CS2/CSGO

> Складено до фінальних обчислень. Фіксує дані, змінні, тести, припущення, ризики, очікувані
> таблиці/фігури. Спирається на аудит article3 (`outputs/logs/article3_audit.md`).

## 1. Дослідницьке питання

Які метрики продуктивності гравця є **інваріантними** (контекстно-незалежними) щодо:
(а) карти, (б) версії гри (CS2 vs CSGO), (в) сторони (T vs CT)?
Які ознаки інваріантні **одночасно за всіма вимірами** — тобто відображають внутрішній скіл гравця,
придатний для моделювання скілу та скаутингу?

## 2. Дані

- **Джерело:** HLTV.org, статистика гравців по картах.
- **Файли (основа v2):** `data/raw/map_stats_cs2_clean.csv` (1852 рядки),
  `data/raw/map_stats_csgo_clean.csv` (1705). Сирі (з `"-"`) — лише для звірки.
- **Одиниця спостереження:** гравець × карта × версія.
- **Карти (8):** de_train, de_nuke, de_inferno, de_mirage, de_dust2, de_vertigo, de_ancient, de_anubis.
- **Ознаки:** ~46 базових метрик (overall) + їх CT-/T-варіанти; категорії: firepower, trading,
  opening, clutching, sniping, utility, support.
- **Період:** CS2 2023-09-27…2025-12-31; CSGO 2012-08-21…2023-09-26.

## 3. Очищення (відтворюване, у scripts/clean_data.py → data/processed/)

1. Конкатенація CS2+CSGO; мітка `game_version`.
2. Фільтр карт: лише `map` що починається з `de_`; відкинути рядки без `game_version`.
3. **`pd.to_numeric(errors='coerce')` для ВСІХ стат-колонок** (усуває дефект типу даних — критично).
4. Виключити метадані: `full_url, player_name, game_version, map, age` та `*time_alive_per_round`
   (текстовий формат "1m 6s"; за потреби — конвертувати в секунди окремо).
5. Зафіксувати n до/після кожного кроку в `outputs/logs/`.

## 4. Статистичні тести (за обґрунтуванням article3, методологічно валідні)

| Вимір | Групи | Структура | Тест | Розмір ефекту | Поріг інваріантності |
|-------|-------|-----------|------|---------------|----------------------|
| Карти | 8 | незалежні | One-way ANOVA (F) | η² | η² < 0.06 |
| Версії | 2 | незалежні | Mann–Whitney U | Cohen's d | d < 0.5 |
| Сторони | 2 | парні (той самий гравець) | Парний t-тест | Cohen's d (парний) | d < 0.5 |

**Зниження розмірності:** PCA (2 компоненти) на стандартизованих overall-ознаках — візуалізація
роздільності груп (карти / версії).

### Принципові рішення (зафіксовано)
- **Інваріантність визначаємо ЛИШЕ за розміром ефекту**, не за p-value (n≈3486 → p майже завжди <0.05).
  Це усуває проблему 2 з аудиту. Фігури кодують інваріантність тим самим критерієм.
- p-values наводимо довідково, але не як критерій.
- Map-ANOVA пояснити з урахуванням, що версія — потенційний конфаундер; додатково перевірити
  інваріантність по картах окремо в межах кожної версії (sensitivity check).

## 5. Очікувані таблиці (outputs/tables/)

- `map_invariance.csv` — feature, F, p, η², invariant(η²<0.06).
- `version_invariance.csv` — feature, cs2_mean, csgo_mean, diff, U-p, Cohen's d, invariant.
- `side_invariance.csv` — feature, ct_mean, t_mean, diff, t, p, Cohen's d(парний), invariant.
- `invariant_features_summary.csv` — переліки + множинні перетини (map∩version, triple).

## 6. Очікувані фігури (outputs/figures/)

- `invariance_summary.png` — 3 панелі ефектів (кодування за η²/d-порогами).
- `pca_projection.png` — PCA по картах і версіях.
- `side_comparison.png` — CT vs T для топ-ознак.
- `feature_distributions.png` — розподіли CS2 vs CSGO для ключових метрик.

## 7. Ризики та припущення

- `_clean.csv` конвертує `"-"` → NaN; перевірити, що NaN не зміщує середні (частка NaN на ознаку логувати).
- Незбалансовані групи карт за кількістю спостережень → ANOVA робастна, але зафіксувати n по групах.
- Снайперські/utility-метрики мають багато нулів/викидів → саме тому Mann–Whitney для версій.
- Період CS2 коротший → можливий зсув складу гравців; обговорити як обмеження.
- Після фіксу даних кількості інваріантних ознак, найімовірніше, **відрізнятимуться від article3** —
  це очікувано; усі числа братимуться з нових outputs/, не з README article3.

## 8. Критерій готовності аналізу

Усі 4 таблиці + 4 фігури згенеровані відтворюваним скриптом; n на кожному кроці залоговано;
переліки інваріантних ознак мають provenance; sensitivity-check по версіях виконано.
