# Стаття 3 (подання в Луцьк) — інструкції проєкту

Відтворюваний конвеєр написання наукової статті. **Це третя стаття** серійної публікаційної лінії.
**Тема:** інваріантність ознак продуктивності гравців CS2/CSGO (карти, версії гри, сторони).
**Цільовий журнал:** №85 «Комп'ютерно-інтегровані технології» (ЛНТУ, Луцьк) — див. `paper/journal_requirements.md`.
**Найближчий дедлайн подання:** 20.08.2026 (вересневий випуск).

## Залізні правила (non-negotiable)

- Ніколи не вигадувати статистичні результати: p-values, розміри ефектів, обсяги вибірок, дати, відсотки.
- Кожне числове твердження в Results має мати provenance: `outputs/tables/`, `outputs/figures/` або `outputs/logs/`.
- Якщо значення немає — писати «not available» і позначати як невирішене.
- Не редагувати файли в `data/raw/`. Похідні файли — лише в `data/processed/`.
- Використовувати plan mode перед великим аналізом, реструктуризацією рукопису, роботою з вимогами журналу.
- Пряме виконання — лише для дрібних, чітко окреслених правок.
- Тримати `review/revision_log.md` актуальним для кожної суттєвої зміни.
- Зберігати посилання та підписи фігур/таблиць.
- Розділяти результати й інтерпретацію; уникати необґрунтованих причинних тверджень.

## Уроки аудиту article3 (обов'язкові для v2 — див. outputs/logs/article3_audit.md)

- **Основа даних:** використовувати `map_stats_cs2_clean.csv` / `map_stats_csgo_clean.csv`
  (сирі файли мають токени `"-"`, через які ~22% ознак мовчки випадали з аналізу).
- При завантаженні застосовувати `pd.to_numeric(col, errors='coerce')` до всіх стат-колонок.
- **Єдине визначення інваріантності — за РОЗМІРОМ ЕФЕКТУ** (η²<0.06; Cohen's d<0.5), не за p-value.
  Фігури мають кодувати інваріантність тим самим критерієм, що й текст.
- Не глушити попередження (`warnings.filterwarnings('ignore')` — прибрати).
- Map-ANOVA: враховувати, що версія гри є конфаундером (перевіряти й у межах версії).

## Структура проєкту

- `data/raw/` — оригінальні датасети, не редагувати. Основа: 2 файли map_stats (CS2/CSGO).
- `data/processed/` — очищені/похідні датасети.
- `scripts/` — відтворюваний аналіз, побудова фігур, експорт.
- `outputs/tables/` — згенеровані статистичні таблиці.
- `outputs/figures/` — згенеровані фігури.
- `outputs/logs/` — логи аналізу та припущення.
- `paper/` — секції рукопису й фінальні чернетки.
- `references/` — настанови журналу, джерела, бібліографія.
- `review/` — коментарі рецензента, revision log, summary для наукового керівника.

## Правила письма

- Академічний тон; чіткі стислі абзаци; не перебільшувати висновки.
- Жодних тверджень без цитат або внутрішнього provenance результату.
- Abstract / Methods / Results / Discussion, фігури й таблиці узгоджені з `paper/journal_requirements.md`.
- Мова: основний текст українською; обов'язкові англомовні метадані (назва, анотація, ключові слова).

## Правила аналізу

- Скрипти над ручними обчисленнями. Кожен артефакт — у `outputs/`.
- Припущення документувати в `paper/analysis_plan.md` або `outputs/logs/`.
- Якщо аналіз залежить від вибору — пояснити вибір до фінальних обчислень.
